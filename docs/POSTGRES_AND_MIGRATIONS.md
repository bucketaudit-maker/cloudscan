# PostgreSQL + Migrations Plan for Production

This document outlines how to move from SQLite to PostgreSQL and introduce versioned schema migrations.

---

## 1. Current state

- **Backend**: Raw `sqlite3`, no ORM. Connection via `get_db()` context manager; all SQL uses `?` placeholders; `sqlite3.Row` for dict-like rows.
- **Schema**: Single `SCHEMA` string in `database.py` with `CREATE TABLE IF NOT EXISTS`; `init_db()` creates tables and seeds providers. No migration history.
- **SQLite-specific usage**:
  - `AUTOINCREMENT` → Postgres: `SERIAL` / `GENERATED BY DEFAULT AS IDENTITY`
  - `datetime('now')` → Postgres: `(now() AT TIME ZONE 'utc')` or `CURRENT_TIMESTAMP`
  - `CREATE VIRTUAL TABLE ... fts5(...)` → Postgres: `tsvector` + GIN index (or `pg_trgm` for LIKE)
  - `files_fts MATCH ?` → Postgres: `to_tsvector(...) @@ plainto_tsquery(...)` or `to_tsquery`
  - `INSERT OR IGNORE` → Postgres: `INSERT ... ON CONFLICT DO NOTHING`
  - `ON CONFLICT(...) DO UPDATE` → Postgres: same idea, syntax `ON CONFLICT (c1,c2) DO UPDATE SET ...`
  - `RANDOM()` → Postgres: `random()` (same name)
  - `cursor.lastrowid` / `cursor.rowcount` → Postgres: `RETURNING id` and `cursor.rowcount`

---

## 2. Goals

1. **Support both SQLite (dev) and PostgreSQL (prod)** driven by `DATABASE_URL` (e.g. `postgresql://user:pass@host:5432/dbname`).
2. **Single code path**: one set of Store classes, with a thin abstraction over connection and placeholder style.
3. **Versioned migrations** for Postgres (and optionally SQLite) so schema changes are repeatable and reversible.
4. **Full-text search** on Postgres with behavior similar to current FTS5 (filepath, filename, extension, content_type).

---

## 3. Phase 1: Connection and placeholder abstraction

**Goal**: `get_db()` and SQL execution work the same for SQLite and Postgres.

1. **Add dependency**
   - `psycopg2-binary` (or `psycopg[binary]` for psycopg 3) in `backend/requirements.txt`.

2. **Config**
   - Already have `DATABASE_URL`. Add a helper, e.g. `settings.is_postgres` → `DATABASE_URL.startswith("postgresql")`.

3. **Connection layer** (`backend/app/models/database.py` or new `backend/app/models/connection.py`)
   - **SQLite**: keep current `sqlite3.connect(...)`, `row_factory=sqlite3.Row`, placeholders `?`.
   - **Postgres**: `psycopg2.connect(settings.DATABASE_URL)`, cursor that returns dict-like rows (e.g. `RealDictCursor` or custom row factory), placeholders `%s`.
   - **Abstraction**: one `get_db()` that yields a connection adapter with:
     - `execute(sql, params)`, `fetchone()`, `fetchall()`, `executemany(...)`
     - Placeholder normalization: either use `%s` everywhere and convert when using SQLite (replace `%s` → `?` for sqlite3), or keep two SQL variants (not ideal). **Recommended**: use a single placeholder style in code (e.g. `%s`) and in the SQLite path replace `%s` with `?` before executing, so all Store code is DB-agnostic.
   - **Commit/rollback**: same as now (commit on success, rollback on exception, close in finally).

4. **Cursor semantics**
   - `lastrowid`: for Postgres use `RETURNING id` in INSERT and read from fetchone().
   - `rowcount`: both drivers support it; keep using it where you do today (e.g. insert_batch).

Implement a small **DB adapter** used by all stores (e.g. `db.execute(sql, params)`, `db.fetchone()`, `db.last_insert_id()` that for Postgres comes from RETURNING). Stores then call this adapter instead of raw connection so SQL can stay as close as possible to one form (e.g. `%s` and Postgres-style SQL by default, with a SQLite compatibility shim that rewrites placeholders and runs SQLite-specific DDL).

## 4. Phase 2: PostgreSQL schema and full-text search

**Goal**: Equivalent schema and search behavior on Postgres.

1. **Table DDL (Postgres)**  
   Create a separate schema definition (or first migration) that uses:
   - `SERIAL` / `BIGSERIAL` or `GENERATED BY DEFAULT AS IDENTITY` for primary keys.
   - `TEXT`, `INTEGER`, `BOOLEAN` (with `true`/`false`).
   - `TIMESTAMPTZ` or `TEXT` for timestamps (if you keep TEXT for compatibility, keep ISO strings).
   - Defaults: `DEFAULT (now() AT TIME ZONE 'utc')` or keep ISO strings in TEXT columns.
   - Unique constraints: `UNIQUE(provider_id, name, region)` etc. — same as now.
   - No virtual tables; for FTS see below.

2. **Full-text search (Postgres)**
   - Add a `tsvector` column to `files`, e.g. `search_vector tsvector GENERATED ALWAYS AS (to_tsvector('english', coalesce(filepath,'') || ' ' || coalesce(filename,'') || ' ' || coalesce(extension,'') || ' ' || coalesce(content_type,''))) STORED`.
   - Create index: `CREATE INDEX idx_files_search ON files USING GIN (search_vector);`
   - Search: replace `files_fts MATCH ?` with `search_vector @@ plainto_tsquery('english', %s)` (and optionally `websearch_to_tsquery` for more control). Keep the same query-sanitization logic you have (terms, NOT, etc.) and map to `plainto_tsquery` or a safe `to_tsquery` expression.
   - For “relevance” ordering you can use `ts_rank(search_vector, query)` in ORDER BY.

3. **Upsert and conflict**
   - Buckets: `INSERT INTO buckets (...) VALUES (...) ON CONFLICT (provider_id, name, region) DO UPDATE SET ...` — same idea as now.
   - Files: `INSERT INTO files (...) VALUES (...) ON CONFLICT (bucket_id, filepath) DO NOTHING` (or DO UPDATE if you want), and use `RETURNING id` only when you need the id.

4. **Seeding**
   - Providers: `INSERT INTO providers (...) VALUES (...) ON CONFLICT (id) DO NOTHING` (or ON CONFLICT (name) DO NOTHING depending on your unique key). Keep seed data in a script or in a migration; run after migrations.

## 5. Phase 3: Migrations (Alembic)

**Goal**: Versioned, repeatable schema changes for production (and optionally dev).

1. **Tool**: **Alembic**.
   - Install: `alembic` in `requirements.txt`.
   - Init: `alembic init backend/alembic` (or `alembic init alembic` at repo root). Prefer under `backend/` if the app is backend-only.

2. **Config**
   - In `alembic.ini` / `env.py`, set `sqlalchemy.url` from `settings.DATABASE_URL` (you can use SQLAlchemy only for the URL and run raw SQL in migrations). For a non-ORM setup, in `env.py`:
     - Use `config.get_main_option("sqlalchemy.url")` or read from your app’s `settings.DATABASE_URL`.
     - For Postgres-only migrations, skip running when `DATABASE_URL` is SQLite (or run a separate SQLite migration path if you want).

3. **Migration style**
   - **Raw SQL migrations**: In each migration’s `upgrade()` use `op.execute("...")` with multi-statement SQL if needed. This keeps one source of truth in SQL and avoids maintaining ORM models.
   - **Initial migration**: One migration that creates all tables, indexes, and the `files.search_vector` column + GIN index for Postgres. No `CREATE IF NOT EXISTS` in migrations; use `IF NOT EXISTS` only where Alembic doesn’t track it (e.g. optional extensions).

4. **Workflow**
   - New change: `alembic revision -m "add_foo_column"` → edit `upgrade()` and `downgrade()`.
   - Deploy: `alembic upgrade head`.
   - Optionally: after migrations, run provider (and demo) seed if not already in a migration.

5. **Downgrades**
   - Implement `downgrade()` for critical migrations (e.g. drop added columns/tables). Full downgrade from scratch is “drop all and re-run from zero” if you don’t need fine-grained rollback.

---

## 6. Phase 4: Wire app to use Postgres when configured

**Goal**: No code paths that assume SQLite-only.

1. **`init_db()`**
   - If Postgres: **do not** run the current `SCHEMA` string (SQLite DDL). Either:
     - Run migrations only (`alembic upgrade head`), and ensure that’s done at deploy time before app start, or
     - From the app, call a small “ensure migrated” that runs `alembic upgrade head` (or a single function that runs the same logic). Prefer migrations at deploy and `init_db()` only opens a connection / runs minimal checks (e.g. “providers exist”).
   - If SQLite: keep current behavior (create from `SCHEMA` if you want), or also use migrations for SQLite for consistency (optional).

2. **Seed**
   - Providers: run from a migration (Postgres) or from `init_db()` (SQLite), or from a single seed script that uses the same `get_db()` and INSERTs with ON CONFLICT so it’s safe on both.

3. **Tests**
   - Keep using SQLite (in-memory or temp file) for speed; use the same adapter so placeholder and connection behavior match. For CI, you can add one job that runs tests against Postgres if desired.

## 7. Suggested implementation order

| Step | Task |
|------|------|
| 1 | Add `psycopg2-binary` and `alembic` to `requirements.txt`. |
| 2 | Introduce a small DB adapter in `database.py`: `get_db()` yields an adapter that normalizes placeholders (`%s` → `?` for SQLite), exposes `execute`, `fetchone`, `fetchall`, `executemany`, `last_insert_id`, `rowcount`. Keep SQL in stores using `%s` and Postgres-compatible syntax where it differs (or add a “dialect” that rewrites DDL for SQLite). |
| 3 | Add `is_postgres` (and optionally `dialect`) to config from `DATABASE_URL`. |
| 4 | Implement Postgres connection in the adapter (RealDictCursor or dict-like rows). |
| 5 | Write **initial Alembic migration** (Postgres-only): create all tables, indexes, and `files.search_vector` + GIN index. Include provider seed in migration or a separate “seed” migration. |
| 6 | Change **FileStore.search** to use `search_vector @@ plainto_tsquery(...)` when `is_postgres`, and keep FTS5 path for SQLite. (Or use adapter so one SQL variant is chosen by dialect.) |
| 7 | Replace SQLite-only DDL in `init_db()`: when Postgres, skip table creation (migrations only); when SQLite, keep current SCHEMA execution or replace with a SQLite migration. |
| 8 | Update **BucketStore.upsert**, **FileStore.insert_batch**, and any **INSERT/ON CONFLICT** to use Postgres-compatible SQL and `RETURNING id` where you need lastrowid. Use adapter’s placeholder style. |
| 9 | Update **routes** that use `RANDOM()` to `random()` (or keep a dialect-specific snippet). |
| 10 | Document in README or DEPLOYMENT: set `DATABASE_URL=postgresql://...`, run `alembic upgrade head` before starting the app. |

## 9. Quick reference: SQL differences

| SQLite | PostgreSQL |
|--------|------------|
| `?` placeholder | `%s` (use adapter to convert for SQLite) |
| `INTEGER PRIMARY KEY AUTOINCREMENT` | `BIGSERIAL PRIMARY KEY` or `... GENERATED BY DEFAULT AS IDENTITY` |
| `datetime('now')` | `(now() AT TIME ZONE 'utc')` or keep TEXT + app default |
| `CREATE VIRTUAL TABLE fts5(...)` | `tsvector` column + GIN index |
| `files_fts MATCH ?` | `search_vector @@ plainto_tsquery('english', %s)` |
| `INSERT OR IGNORE` | `INSERT ... ON CONFLICT DO NOTHING` |
| `cursor.lastrowid` | `RETURNING id` + fetchone() |
| `RANDOM()` | `random()` |

This plan keeps your existing Store design and raw SQL, adds a thin connection/placeholder layer, introduces Postgres and full-text search, and uses Alembic for production migrations.
